# Commute-Home — 통근 기회비용 기반 동네 추천 서비스 설계

**작성일:** 2026-06-24
**상태:** 설계 합의 완료 (구현 계획 작성 전)
**작업 폴더:** `~/Documents/commute-home`

---

## 1. 제품 정의

### 한 줄 정의
> 직장 위치·예산·통근수단·취향을 입력하면, **실거래가와 통근 기회비용까지 계산**해 "너에게 최적인 동네 Top N"을 점수와 이유로 추천하고, 네이버부동산 매물 검색으로 연결하는 SaaS.

### 목적
- 정식 SaaS (포트폴리오 + 수익화).
- 선정(Seonjung) 프로젝트와 동일하게 포트폴리오 가치 + 실수익을 함께 노린다.

### 핵심 가치 / 차별점
- 부동산 탐색은 **① 탐색·의사결정** 단계와 **② 거래(계약)** 단계로 나뉜다. 본 서비스는 **①을 책임지고 ②로 자연스럽게 연결**한다.
- 기존 분석 도구(호갱노노·아실·부동산지인)는 **매매·투자** 중심이며, **월세 세입자의 "통근 기회비용"** 은 아무도 제대로 다루지 않는다 → **여기가 차별점.**
- 킥이 되는 분석: **총비용 관점**
  > "A동은 월세 5만원 싸지만, 통근이 왕복 40분 더 길어 시간가치 환산 시 월 18만원 손해. 결국 B동이 월 13만원 이득."

---

## 2. 핵심 제약 (의사결정 근거)

한국 부동산은 **"현재 계약 가능한 매물"** 을 주는 OPEN API가 사실상 없다.
- 네이버부동산 / 직방 / 다방 → 공식 OPEN API 미제공 (크롤링은 약관 위반·법적 리스크).
- 국토교통부 실거래가 API → **과거 거래 내역만** 제공 (실시간 매물 ❌).

**결정:** 매물 직접 제공을 포기하고, 제품을 **"실거래가 기반 지역/조건 추천 + 네이버부동산 딥링크 연결"** 로 재정의한다.
- 합법적 (딥링크는 약관 위반 아님).
- 데이터 전부 확보 가능.
- 차별점 명확.

## 3. 시스템 구조 (방식 A: 점수엔진 + LLM 설명)

```
[사용자 입력]
  직장주소 · 월예산(보증금/월세) · 통근수단 · 시간가치(시급) · 취향가중치
        │
        ▼
[데이터 수집 레이어]  ← OPEN API
  ├ 국토부 실거래가 API   → 동네별 오피스텔/원룸 월세 시세
  ├ 카카오 로컬 API       → 후보 동네 좌표 + 주변 편의시설
  ├ 카카오/대중교통 API   → 직장까지 거리·소요시간 (차/대중교통)
  └ 오피넷 유가 API        → 지역 유가 → 유류비 계산
        │
        ▼
[점수 엔진]  (결정론적·수식, 순수 TS 모듈)
  각 후보 동네마다 5개 축 점수 계산 → 가중합 → 랭킹
        │
        ▼
[LLM 설명 레이어]  (Claude)
  Top N 동네를 "왜 추천하는지" 자연어로 설명 + 비교
        │
        ▼
[결과 화면]  동네 랭킹 + 점수 분해 + [네이버부동산에서 매물 보기] 딥링크
```

### 아키텍처 원칙
- **결정론적 엔진이 정확도를 책임진다.** 비용·거리·시세 계산은 전부 수식(순수 TS 모듈)으로, 테스트 가능하고 재현 가능하게.
- **LLM은 가장 잘하는 일만 한다.** 자연어 설명·비교·취향 반영. 숫자 계산을 LLM에 맡기지 않는다 (환각 방지).
- 방식 B(LLM이 전부 추론)는 비용·속도·숫자 환각·비재현성 때문에 채택하지 않음.

## 4. 점수 엔진 — 5개 축 (핵심 IP)

각 후보 동네에 대해 5개 축 점수를 계산하고 사용자 가중치로 가중합하여 랭킹한다.

| 축 | 계산 근거 | 데이터 소스 |
|---|---|---|
| **① 주거비** | 동네 평균 월세가 예산 대비 얼마나 여유 있는가 | 국토부 실거래가 |
| **② 통근 실비용** | 월 유류비(차) 또는 교통비(대중교통), 왕복 × 근무일 | 거리×유가 / 대중교통 요금 |
| **③ 통근 기회비용** | 월 통근시간 × 본인 시간가치(시급) — **차별점** | 소요시간 API |
| **④ 편의시설** | 반경 내 편의점·헬스장·병원·마트 밀도 | 카카오 로컬 |
| **⑤ 삶의질 취향** | 사용자가 정한 우선순위(조용함/번화가 등) 가중치 | 사용자 입력 |

### 총비용(월 환산) 모델 — 제품의 킥
- **월 총비용 = 월세 + 통근 실비용(②) + 통근 기회비용(③)**
- 시간가치(시급)는 사용자가 직접 설정하며, 합리적 기본값을 제공한다 (예: 본인 추정 시급, 미입력 시 최저임금 또는 사용자 평균 환산값).
- 결과 카드에 "월 환산 총비용"을 명시해, 단순 월세 비교가 아닌 **실질 부담 비교**를 보여준다.

## 5. 기술 스택

- **프론트/백:** Next.js (App Router) + TypeScript — 웹 SaaS
- **DB/Auth/결제연동:** Supabase (후보 동네·시세 캐시·사용자 데이터)
- **점수 엔진:** 순수 TypeScript 모듈 (서버 실행, 단위 테스트 대상)
- **LLM:** Claude (Anthropic API) — 설명·대화만 담당
- **OPEN API:** 국토부 실거래가 / 카카오 로컬·길찾기 / 오피넷 유가
- **배포:** Vercel

---

## 6. 화면 (MVP — 3개)

1. **입력 화면** — 직장주소, 예산(보증금/월세), 통근수단(자동차/대중교통), 시간가치(시급, 기본값 제공), 취향 슬라이더 3~4개.
2. **결과 화면** — 동네 Top 5 카드(종합점수 + 5축 막대 + "월 환산 총비용") + LLM 설명 + [네이버부동산에서 보기] 딥링크.
3. **상세 비교** — 동네 2~3곳 나란히 비교 ("A vs B 총비용").

---

## 7. MVP 범위 (명확한 경계)

### 포함 (✅)
- 월세 오피스텔/원룸, **특정 지역 1곳**부터 시작 (예: 서울 일부 권역 또는 본인 직장 권역).
- 통근수단: **자동차 + 대중교통 둘 다**.
- 5축 점수 엔진 + LLM 설명 + 네이버부동산 딥링크.
- 3개 화면 (입력 / 결과 / 상세 비교).

### 제외 — 다음 버전 (❌)
- 전세 / 매매 (거래유형 확장).
- 실시간 매물 데이터.
- 제휴 수수료 정산.
- 알림 / 저장 자동화.
- 도보·자전거 통근수단.

---

## 8. 수익 모델 (방향만, MVP 구현 아님)

- 기본 추천 무료 → **상세 비교·저장·리포트는 유료**(프리미엄 구독).
- 장기: 네이버부동산/제휴 **딥링크 전환 수수료**.

---

## 9. 합법성 / 리스크 메모

- 매물 크롤링은 하지 않는다. 공공 OPEN API + 사용자가 따라가는 딥링크만 사용.
- 딥링크는 검색 조건이 미리 채워진 외부 링크일 뿐이라 약관 위반이 아니다.
- 실거래가는 "과거 평균 시세" 참고용임을 UI에 명시 (실시간 매물가와 다를 수 있음).

---

## 10. 열린 질문 (구현 계획 단계에서 확정)

- 시작 지역(특정 1곳)을 어디로 할지 — 본인 직장 권역 vs 데이터 풍부한 강남권.
- "후보 동네"의 단위 — 법정동 / 행정동 / 지하철역 반경 중 무엇으로 묶을지.
- 시간가치(시급) 기본값 산정 방식의 구체 수치.
- 5개 축 가중치의 초기 기본값.
- 대중교통 소요시간 API 선택 (카카오 vs 별도 대중교통 API) 및 무료 호출 한도.
